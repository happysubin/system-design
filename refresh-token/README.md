# 리프레시 토큰 시스템

## 개요

이 프로젝트는 다음 범위를 기준으로 한 간단한 토큰 기반 인증 흐름을 구현합니다.

- **액세스 토큰**: 15분
- **리프레시 토큰**: 14일
- **액세스 토큰 상태**: 서버에 저장하지 않음
- **리프레시 토큰 상태**: 캐시에 저장
- **포함 API**:
  - 로그인 API
  - 토큰 재발급 API
  - 로그아웃 API

이 버전은 가벼운 운영 모델을 목표로 합니다. 액세스 토큰은 stateless하게 유지하고, 리프레시 토큰만 서버에서 관리합니다.

또한 Redis 장애 시에는 **degraded mode** 로 동작합니다. 이 경우 로그인은 access token만 발급하여 허용하고, refresh token 재발급은 실패 처리하여 다시 로그인하도록 유도합니다.

현재 구현은 별도 회원 DB 없이 **설정 기반 demo 사용자**를 사용합니다.

- username: `demo`
- password: `password`

## 인증 정책

### 1. 액세스 토큰

- 로그인 성공 후 발급
- **15분** 동안 유효
- 인증이 필요한 API 요청에 사용
- 토큰 검증 로직으로만 확인
- **DB나 캐시에 저장하지 않음**

목표는 일반 인증 요청에서 서버 측 조회를 없애고 빠르게 처리하는 것입니다.

### 2. 리프레시 토큰

- 로그인 시 액세스 토큰과 함께 발급
- **14일** 동안 유효
- 새로운 액세스 토큰 발급 시에만 사용
- 만료 시간을 가진 상태로 **캐시에 저장**
- 재발급 시 새 리프레시 토큰으로 교체됨
- 로그아웃 시 삭제

이 설계에서 리프레시 토큰은 유일한 서버 관리 세션 상태입니다.

## 저장 전략

### 액세스 토큰

- Stateless
- 영속 저장 없음
- 캐시 조회 없음
- 만료 여부는 토큰 클레임으로 판별
- 블랙리스트를 따로 두지 않음

이 설계에서는 access token을 서버에 저장하거나 블랙리스트로 관리하지 않습니다. 대신 access token 만료 시간을 15분으로 짧게 가져가서 노출 위험 시간을 줄이고, 상태 관리가 필요한 인증 제어는 refresh token 쪽에서 처리합니다.

### 리프레시 토큰

- **Redis** 같은 캐시에 저장
- 캐시 엔트리 TTL은 리프레시 토큰 만료 시간과 동일하게 설정
- 캐시는 리프레시 토큰 검증과 로그아웃 무효화의 기준 저장소 역할을 함
- Redis 키는 환경과 애플리케이션을 포함한 네임스페이스를 사용함

예시 개념 데이터:

```text
key: {env}:{app}:auth:refresh-token:v1:{sha256(refreshToken)}
value: userId / session metadata
ttl: 14 days
```

예시:

```text
local:refresh-token:auth:refresh-token:v1:<sha256>
test:refresh-token:auth:refresh-token:v1:<sha256>
prod:refresh-token:auth:refresh-token:v1:<sha256>
```

이 방식은 같은 Redis 인스턴스를 여러 환경이나 여러 애플리케이션이 공유하더라도 키 충돌을 줄이고, 운영 시 스캔/삭제/모니터링 범위를 명확하게 나누기 위한 설계입니다.

## API 요구사항

### 1. 로그인 API

`POST /api/auth/login`

#### 목적

사용자를 인증하고 두 종류의 토큰을 발급합니다.

#### 동작

1. 사용자 자격 증명을 검증한다
2. 15분짜리 액세스 토큰을 발급한다
3. 14일짜리 리프레시 토큰을 발급한다
4. 리프레시 토큰 상태를 TTL과 함께 캐시에 저장한다
5. 두 토큰을 클라이언트에 반환한다

#### 기대 결과

- 올바른 자격 증명일 때만 로그인에 성공한다
- 액세스 토큰은 즉시 보호된 API 호출에 사용할 수 있다
- 리프레시 토큰은 이후 액세스 토큰 재발급에 사용할 수 있다
- Redis가 정상일 때는 access token과 refresh token을 모두 발급한다
- Redis 장애 시에는 로그인은 성공하지만 **access token만 발급**하고, 응답에 degraded 상태를 명시한다

### 2. 토큰 재발급 API

`POST /api/auth/refresh`

#### 목적

유효한 리프레시 토큰을 사용해 새로운 토큰 쌍을 발급합니다.

#### 동작

1. 클라이언트로부터 리프레시 토큰을 전달받는다
2. 토큰 형식과 만료 여부를 검증한다
3. 해당 리프레시 토큰이 캐시에 존재하는지 확인한다
4. 유효하면 기존 리프레시 토큰을 무효화한다
5. 새로운 액세스 토큰과 새로운 리프레시 토큰을 발급한다
6. 새 리프레시 토큰 상태를 캐시에 저장한다
7. 새 토큰 쌍을 반환한다

#### 기대 결과

- 리프레시 토큰이 유효하고 캐시에 존재하면 새로운 액세스 토큰과 새로운 리프레시 토큰을 반환한다
- 리프레시 토큰이 만료되었거나 유효하지 않거나 캐시에 없으면 요청은 실패하고 사용자는 다시 로그인해야 한다
- 현재 구현은 **재발급 시 refresh token도 함께 새로 발급**하고, 이전 refresh token은 즉시 무효화한다
- 즉, **refresh token rotation**은 적용했지만, 재사용 시도를 별도 상태로 추적하는 **reuse detection**은 아직 포함하지 않는다
- Redis 장애 시에는 refresh를 허용하지 않고, `503 Service Unavailable`로 실패시켜 재로그인을 유도한다

### 3. 로그아웃 API

`POST /api/auth/logout`

#### 목적

현재 리프레시 토큰 세션을 무효화합니다.

#### 동작

1. 클라이언트로부터 리프레시 토큰 또는 세션 식별자를 전달받는다
2. 해당 리프레시 토큰 상태를 캐시에서 제거한다
3. 로그아웃 성공 응답을 반환한다

#### 기대 결과

- 로그아웃 이후 해당 리프레시 토큰은 더 이상 사용할 수 없다
- 기존 액세스 토큰은 추가 무효화 로직이 없는 한 원래 만료 시점까지는 유효하다
- 현재 방식은 로그아웃 시 리프레시 토큰을 Redis에서 **삭제해서 재사용을 막는 구조**이며, rotation 과정에서도 이전 refresh token을 제거한다
- 다만 재사용 시도를 별도 상태로 추적하는 reuse detection은 아직 포함하지 않는다

## 요청 흐름

### 로그인

```text
Client -> Login API -> 액세스 토큰 + 리프레시 토큰 발급 -> 리프레시 토큰을 캐시에 저장
```

Redis 장애 시:

```text
Client -> Login API -> 액세스 토큰만 발급 -> sessionMode=access_only, degradedAuth=true 반환
```

### 보호된 리소스 접근

```text
Client -> Protected API -> 액세스 토큰만 검증
```

### 재발급

```text
Client -> Refresh API -> 기존 리프레시 토큰 무효화 -> 새 액세스 토큰 + 새 리프레시 토큰 발급 -> 새 리프레시 토큰을 캐시에 저장
```

Redis 장애 시:

```text
Client -> Refresh API -> 503 Service Unavailable -> 다시 로그인 유도
```

### 로그아웃

```text
Client -> Logout API -> 캐시에서 리프레시 토큰 삭제
```

## 이 버전의 특징

- 구현이 단순함
- 일반 인증 요청에서 DB 조회가 없음
- access token은 stateless하게 유지됨
- refresh token rotation이 적용됨
- Redis 장애 시 degraded mode 로그인 지원
- 캐시 삭제만으로 로그아웃 처리 가능
- 캐시 TTL로 만료 관리가 쉬움

## 이 버전의 한계

- 캐시가 유실되면 리프레시 세션도 함께 유실됨
- 캐시 재시작 또는 eviction 이후 사용자가 다시 로그인해야 할 수 있음
- reuse detection은 아직 없음
- 이 README는 단순한 캐시 기반 세션 모델을 설명하며, 완전한 다중 기기 세션 아키텍처는 다루지 않음

## 향후 확장 가능 항목

- 재사용 감지(reuse detection)
- 기기별 세션 관리
- DB 기반의 영속 세션 저장
- 모든 기기 로그아웃
