# payment-lab

PG사를 연동하는 가상 결제 플로우를 설계하고 구현하기 위한 실험용 프로젝트입니다.

## 목표

- 주문과 결제를 분리한 결제 도메인을 설계한다.
- PG 연동 시 발생하는 동기 응답과 비동기 웹훅을 모두 안전하게 처리한다.
- 중복 요청, 중복 승인, 중복 웹훅에도 동일한 결과를 보장하는 멱등성을 확보한다.
- 실패 상황을 사용자 오류, PG 오류, 시스템 오류로 구분해 대응 전략을 명확히 한다.

## 결제 플로우 범위

현재 목표는 특정 결제수단 하나만 하드코딩하는 방식이 아니라, **공통 결제 코어 + PG 연동 + 결제수단별 확장**이 가능한 구조를 준비하는 것입니다.

이 문서는 두 층으로 읽으면 됩니다.

- **현재 구현 기준**: 지금 코드로 실제 동작하는 흐름
- **목표 요구사항**: 이후 단계에서 추가로 구현할 운영형 요구사항

핵심 흐름은 다음과 같습니다.

1. 주문서/외부 주문 정보 준비
2. 주문 생성 시 checkout key 발급
3. 결제 시작 API 호출
4. PG 웹훅 또는 재조회로 최종 상태 확정

## API 호출 순서

현재 기준의 권장 API 순서는 다음과 같습니다.

1. 외부 주문서/주문 시스템이 `orderId`, `merchantOrderId`, `amount`를 준비한다.
   - 결제 서비스는 주문 자체를 만드는 대신, 이미 준비된 주문 정보를 입력으로 받는 구조를 기준으로 한다.

2. `POST /api/v1/payments`
   - 외부 주문 시스템이 넘겨준 `orderId`, `merchantOrderId`, `amount`, `checkoutKey`를 기준으로 결제를 시작한다.
   - 서버는 먼저 checkout key가 Redis에 존재하고 요청 정보와 일치하는지 확인한다.
   - 그 다음 내부 결제 시도를 만들고 바로 PG 승인 요청까지 진행한다.
   - 동일한 checkout key로 다시 요청이 들어오면 기존 결제 시도를 재반환한다.
   - checkout key 하나로 주문서 세션 유효성과 결제 시작 중복 방지를 함께 처리한다.

3. `POST /api/v1/payment-webhooks` 또는 PG 조회 API 연계
   - PG 웹훅 또는 후속 조회를 통해 실제 승인 결과를 확인한다.
   - 현재 구현 기준으로는 `merchantOrderId`, `pgTransactionId`, `secret`, 결과값을 받아 결제 시도를 찾는다.
   - 토스페이먼츠 기준으로는 승인 시 저장한 `secret`과 웹훅 body의 `secret`이 같을 때만 정상 웹훅으로 처리한다.
   - 성공이면 `DONE`, 실패면 `FAILED`, 되돌려야 하면 `CANCELLED`로 확정한다.
   - 웹훅을 최종 확정 경로로 두는 이유는, PG가 비동기적으로 성공/실패를 다시 통지하는 경우가 많기 때문이다.

4. `POST /api/v1/payments/{paymentAttemptId}/reconcile`
   - `PENDING` 상태의 결제 시도를 PG에 다시 조회해 최종 상태를 확정한다.
   - 여기서 `reconcile`은 정산이 아니라, 우리 상태와 PG 상태를 다시 맞춰보는 재확인/재확정 의미다.
   - 조회 결과가 성공이면 `DONE`, 실패면 `FAILED`로 확정한다.
   - 재확정 API를 두는 이유는, 웹훅이 늦거나 누락된 경우에도 서버가 최종 상태를 다시 확인할 수 있어야 하기 때문이다.

간단히 보면 `주문서/외부 주문 준비 → 주문 생성 시 checkout key 발급 → 결제 시작 → 웹훅/조회로 최종 확정` 흐름이다.

## 개발/테스트용 보조 API

- `POST /api/v1/orders`
  - 현재 코드에는 로컬 테스트와 흐름 검증을 위해 주문 생성 API가 남아 있다.
  - 이 API는 `merchantOrderId`, `checkoutKey`를 서버가 발급하고 `amount`와 함께 저장한다.
  - 하지만 권장 아키텍처에서는 메인 결제 흐름의 시작점이 아니라, 외부 주문 시스템을 대신하는 보조 API로 본다.

## 현재 구현 의도

- `merchantOrderId`는 주문 기준 식별자다. 최종 목표는 외부 주문 시스템이 이를 만들고, 결제 서비스는 그 값을 받아 PG 승인 요청, 조회, 취소, 웹훅 대사에 재사용하는 구조다.
- `checkoutKey`는 주문 생성 또는 주문서 준비 시점에 함께 발급되는 짧은 생명주기의 키다. Redis TTL로 관리하고, 실제 결제 시작 시점에만 유효성을 검증한다.
- 현재 구현에서는 `checkoutKey` 하나가 주문서 세션 유효성과 결제 시작 중복 방지를 함께 맡는다.
- 결제 시작 API 하나가 내부적으로 결제 시도 생성과 PG 승인 요청을 연속 수행한다.
- 결제 승인 직후 상태를 `DONE`으로 두지 않고 `PENDING`으로 두는 이유는, 동기 응답과 비동기 웹훅/조회 결과가 어긋날 수 있기 때문이다.
- 웹훅과 재확정(reconcile)을 둘 다 두는 이유는, 웹훅은 실시간성에 강하고 조회는 누락 복구에 강해서 서로 보완 관계이기 때문이다.
- 상태 가드를 둔 이유는 `READY`가 아닌 결제 승인, `PENDING`이 아닌 최종 확정, 이미 끝난 결제의 중복 웹훅 반영을 막기 위해서다.
- checkout key를 별도로 둔 이유는 주문서 진입 자체는 막지 않으면서, 실제 결제하기 중복만 제어하기 위해서다.
- 현재 구현은 분산 락 대신 조건부 상태 전이와 DB 유니크 제약으로 먼저 안정성을 확보하는 방향을 택했다.
- 즉 승인 요청은 `READY -> PENDING`, 최종 확정은 `PENDING -> DONE/FAILED`를 조건부 업데이트로 처리해 동시에 여러 요청이 와도 한 번만 상태가 바뀌게 한다.
- 토스 웹훅은 헤더 서명 대신 payment secret 비교 방식에 맞춰, 결제 승인 시 받은 secret을 저장하고 웹훅 수신 시 다시 검증한다.

## 핵심 요구사항

### 1. 주문과 결제 분리

- 주문(Order)과 결제 시도(PaymentAttempt)는 별도 엔티티로 관리한다.
- 하나의 주문은 여러 번 결제를 시도할 수 있지만, 동시에 활성 상태인 결제 시도는 정책적으로 제한한다.
- 결제 금액, 주문 금액, PG 승인 금액은 항상 비교 가능해야 한다.

### 2. 결제 상태 관리

결제는 명시적인 상태 전이를 가져야 한다.

#### 현재 구현 기준

- `READY`
- `PENDING`
- `DONE`
- `FAILED`
- `CANCELLED`

현재 코드에서는 승인 요청 시 `READY → PENDING`, 웹훅 또는 재확정 시 `PENDING → DONE/FAILED` 흐름을 사용한다.

#### 목표 요구사항

- `AUTH_REQUESTED`
- `EXPIRED`

상태 전이 규칙은 서버가 강제해야 하며, 이미 완료된 결제에 잘못된 상태가 다시 반영되면 안 된다.

- 승인 요청은 `READY` 상태의 결제 시도에만 허용한다.
- 웹훅 또는 후속 조회를 통한 최종 확정은 `PENDING` 상태의 결제 시도에만 허용한다.
- 이미 `CANCELLED` 된 결제 시도는 웹훅이 다시 와도 상태를 바꾸지 않고 무시한다.

### 3. PG 연동 처리

- PG 승인 요청 전 내부 결제 시도를 먼저 생성한다.
- PG의 동기 응답만 신뢰하지 않고, 필요하면 웹훅 또는 후속 조회를 통해 최종 상태를 확정한다.
- PG 거래 식별자(`pgTransactionId` 등)를 내부 결제 시도와 매핑할 수 있어야 한다.

> 현재 구현은 stub PG 클라이언트 기준의 최소 흐름이다. 서명 검증, 실제 네트워크 오류, 운영형 보상 처리까지는 아직 포함하지 않는다.

## 중복 요청 방지 요구사항

중복 방지는 이 프로젝트의 핵심 요구사항이다.

### 1. 사용자 중복 요청 방지

- 같은 주문에 대해 결제 버튼 연속 클릭이 발생해도 한 번만 유효하게 처리되어야 한다.
- 이를 위해 `merchantOrderId`, `paymentAttemptId`, `checkoutKey` 같은 식별자를 사용한다.
- `checkoutKey`는 주문서 세션의 유효성과 결제 시작 시점의 1회성 검증에 사용한다.
- 동일한 `checkoutKey`로 다시 요청이 들어오면 새로운 결제를 만들지 않고 기존 결제 시도를 재반환한다.

### 2. PG 중복 요청/중복 콜백 방지

- PG가 동일 승인 결과를 두 번 보내거나 동일 웹훅을 재전송해도 시스템 상태는 한 번만 변경되어야 한다.
- 동일한 `pgTransactionId` 또는 PG 고유 토큰에 대해 중복 반영을 금지한다.
- 이미 성공 처리된 결제에 동일 성공 이벤트가 다시 와도 부작용 없이 무시되어야 한다.

### 3. 동시성 제어

- DB 레벨 유니크 제약조건으로 최종 방어선을 둔다.
- 상태 변경은 조건부 업데이트 또는 compare-and-set 방식으로 처리한다.
- 동시에 두 요청이 들어와도 최종적으로 하나만 성공 반영되도록 보장해야 한다.

> 위 항목은 현재 목표 요구사항이며, 아직 코드에 완전히 반영되지는 않았다.

## 예외 처리 요구사항

예외는 최소한 아래 범주로 나눠 관리한다.

### 1. 사용자/입력 오류

- 존재하지 않는 주문
- 이미 완료된 주문 재결제 시도
- 금액 불일치
- 허용되지 않은 상태에서의 결제 요청

대응 원칙:

- 사용자에게 명확한 실패 사유를 전달한다.
- 재시도 가능 여부를 구분한다.

### 2. PG 연동 오류

- 승인 거절
- 타임아웃
- 일시적 네트워크 장애
- 웹훅 서명 검증 실패
- PG 응답 데이터 누락 또는 변조

대응 원칙:

- 타임아웃처럼 PG 승인 성공 여부가 불명확한 경우도 존재한다.
- 이런 경우 내부 상태는 즉시 실패나 취소로 확정하지 않고 **`PENDING` 상태**로 둔다.
- 이후 PG 조회 API 또는 웹훅으로 실제 승인 결과를 확인한다.
- 조회 또는 웹훅 결과가 성공이면, 내부 상태도 `PENDING`에서 `DONE`으로 승격한다.
- 단, 이미 취소가 확정되었거나 보상 취소가 실행된 이후에는 단순 조회 결과만으로 다시 성공 처리하지 않는다.
- 최종적으로 승인 성공이 확인되었지만 비즈니스상 결제를 되돌려야 하는 경우에는 그때 PG 취소 API 또는 환불 API를 호출한다.

> 현재 구현은 `reconcile` API로 재조회 기반 확정을 지원하지만, 실제 PG 타임아웃 감지와 자동 재시도 정책은 아직 없다.

### 3. 시스템 오류

- DB 저장 실패
- 트랜잭션 중간 실패
- 내부 상태 전이 불가

대응 원칙:

- 결제 성공 여부가 불명확하면 안전하게 재조회 가능한 구조여야 한다.
- 로그, 추적 ID, 운영 확인 포인트를 남겨야 한다.

## 웹훅 처리 요구사항

- 웹훅은 반드시 서명 검증 또는 PG 제공 검증 수단을 통과해야 한다.
- 웹훅은 멱등적으로 처리되어야 한다.
- 이미 처리한 이벤트인지 식별할 수 있어야 한다.
- 웹훅 수신 순서가 바뀌어도 최종 상태가 깨지지 않아야 한다.
- 이미 `DONE` 또는 `FAILED`로 확정된 결제 시도에 동일 웹훅이 다시 오면 상태를 다시 바꾸지 않고 무시해야 한다.

> 현재 구현은 멱등 처리와 상태 가드는 포함하지만, 웹훅 서명 검증은 아직 구현하지 않았다.

## 비기능 요구사항

- 모든 결제 요청과 응답에 추적 가능한 로그를 남긴다.
- 주문 ID, 결제 시도 ID, PG 거래 ID를 기준으로 조회 가능해야 한다.
- 장애 상황에서 재처리 가능성을 고려해 이벤트 이력 또는 상태 변경 이력을 남긴다.

## 테스트 포인트

최소한 아래 시나리오는 검증 대상에 포함한다.

1. 정상 승인 성공
2. 동일 checkoutKey로 중복 결제 시작 요청
3. 같은 주문에 대한 동시 결제 요청
4. 동일 웹훅 재전송
5. 승인 성공 후 웹훅 지연 도착
6. PG 타임아웃 후 조회로 성공 확정
7. 금액 불일치 실패
8. 이미 완료된 결제의 중복 성공 콜백 무시
9. PG 타임아웃 후 조회 결과 실패 확정
10. 미확정 상태에서 성공 확정 후 필요 시 보상 취소 수행
